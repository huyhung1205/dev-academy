Workflow cơ bản và xem lịch sử thay đổi

Bài này là trái tim của Git: bạn sẽ hiểu file đi qua các vùng làm việc nào, cách tạo commit đúng và cách đọc lại lịch sử thay đổi.

Video mô phỏng

Video mô phỏng sẽ được cập nhật sau.

Mục tiêu học tập

  • Phân biệt working directory, staging area và local repository
  • Dùng git status, git add, git commit đúng quy trình
  • Đọc git log, git diff và git show để kiểm tra thay đổi

Tài liệu lý thuyết

Working directory, staging area, local repository

Working directory là nơi bạn sửa file, staging area là vùng chờ để chọn phần sẽ commit, còn local repository là nơi Git lưu snapshot sau khi commit.

git status, git add, git commit

Workflow chuẩn là sửa file, kiểm tra trạng thái bằng git status, đưa thay đổi vào staging bằng git add rồi tạo commit bằng git commit -m.

git log, git diff, git show

Dùng git log để đọc lịch sử, git diff để so sánh thay đổi và git show để xem chi tiết một commit cụ thể.

Làm việc có kỷ luật

Commit nhỏ, message rõ ràng và luôn kiểm tra trạng thái trước khi commit giúp lịch sử sạch, dễ đọc và dễ sửa lỗi hơn.

Workflow thực tế

  • Sửa file trong working directory.
  • Kiểm tra bằng git status, stage bằng git add, commit bằng git commit.
  • Đọc lại git log và git diff để xác nhận thay đổi.

Lỗi thường gặp và cách khắc phục

  • Commit khi chưa kiểm tra file đã stage hay chưa.
  • Dùng message quá chung chung như “update” hoặc “fix”.
  • So sánh sai phạm vi khi đọc git diff.

Ví dụ minh họa

Ví dụ 1: Tạo commit đầu tiên

Lưu một thay đổi nhỏ vào lịch sử Git.

git status
git add index.html
git commit -m "Add homepage heading"

Output mẫu

[main abc1234] Add homepage heading

Giải thích lệnh

  • Mục đích: stage file đã sửa và tạo commit có ý nghĩa.
  • Khi dùng: sau khi hoàn thành một nhóm thay đổi nhỏ.
  • Kết quả mong đợi: Git tạo commit mới và in ra mã ngắn của commit.
  • Lỗi thường gặp: quên add file nên commit xong không thấy thay đổi.

Ví dụ 2: Xem lịch sử và so sánh

Kiểm tra lại thay đổi trước khi tiếp tục làm việc.

git log --oneline --graph -5
git diff
git show a1b2c3d

Output mẫu

* a1b2c3d Add header

Giải thích lệnh

  • Mục đích: đọc lịch sử và xem phần thay đổi chi tiết.
  • Khi dùng: khi debug hoặc trước khi gửi code cho người khác review.
  • Kết quả mong đợi: thấy commit, diff và nội dung thay đổi rõ ràng.
  • Lỗi thường gặp: log quá dài nếu không dùng tuỳ chọn gọn hơn.

Bài tập thực hành

Dễ

  • Tạo một commit nhỏ với nội dung rõ ràng.
  • Chạy git status trước và sau khi dùng git add.
  • Dùng git log --oneline để xem commit vừa tạo.

Trung bình

  • Tạo hai thay đổi khác nhau và commit tách biệt.
  • Dùng git diff để so sánh file đang sửa với commit gần nhất.

Khó: Thử đọc lịch sử bằng git log --oneline --graph --decorate và giải thích từng dòng.

Câu hỏi tự luận tự ôn tập

1. Hãy giải thích 3 vùng làm việc của Git bằng ví dụ thực tế.

Working directory là nơi sửa file, staging area là nơi chọn phần sẽ commit, còn local repository là nơi lưu snapshot cuối cùng sau khi commit.

2. Vì sao nên commit nhỏ và có message rõ ràng?

Commit nhỏ giúp đọc lịch sử dễ hơn, tìm lỗi nhanh hơn và giảm rủi ro khi phải quay lại một mốc cũ.
